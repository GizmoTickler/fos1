# Event-Correlation Runtime Design

Date: 2026-04-20
Status: Approved for planning
Scope: EC-1 repo-owned event-correlator runtime

## Context

The repository now has a repo-owned observability proof for:

- Suricata canary ingestion into Elasticsearch with ILM/template verification
- Prometheus pod-annotation scraping for node-local `dpi-manager` and `ntp-controller`

The next repo-owned gap is event correlation runtime depth. The current `EventCorrelation` API and controller can generate config and reconcile a Deployment and Service, but the runtime contract is still incomplete:

- the controller points at `fos1/event-correlator:latest` without a repo-owned image build path
- the runtime is file-based, but the Deployment does not mount a repo-owned source path into the pod
- the current sink mount is only an `EmptyDir`, which is insufficient to describe a durable repo-owned input/output contract

This design keeps the next slice intentionally narrow. It makes the current event-correlation API buildable and runnable without expanding scope into Elasticsearch sinks, distributed event streaming, or threat-intelligence features.

## Goals

- Keep the existing `EventCorrelation` API centered on a small repo-owned contract.
- Make `cmd/event-correlator` a real repo-owned runtime artifact.
- Make the controller generate a Deployment that can actually read its configured file source and write to its configured sink.
- Keep the runtime contract deterministic and testable in unit and controller tests.

## Non-Goals

- No Elasticsearch sink in this slice.
- No message-bus or central fan-in topology.
- No threat-intelligence enrichment.
- No attempt to prove end-to-end live security-event production beyond the bounded file-based correlator contract.
- No redesign of the monitoring or log-shipping stack.

## Recommended Approach

Use a bounded node-local file contract:

- `source.type=file`
- `source.format=jsonl`
- `sink.type=file|stdout`
- `sink.format=json`

This matches the current CRD and runtime code shape and avoids inventing a second correlator contract. The controller becomes responsible for validating and wiring the file paths into the pod. The runtime remains responsible for reading JSON-line input, applying rule threshold/time-window logic, writing correlated output, and exposing probe endpoints.

## Repo-Owned Contract

### API Contract

The `EventCorrelation` API remains minimal:

- `spec.source.type` must be `file`
- `spec.source.path` is the input file path
- `spec.source.format` must be `jsonl` or empty
- `spec.sink.type` must be `file` or `stdout`
- `spec.sink.path` is required when `sink.type=file`
- `spec.sink.format` must be `json` or empty

No new source or sink kinds are introduced in this slice.

### Runtime Contract

The repo-owned runtime is:

- binary: `/usr/bin/event-correlator`
- config path: `/etc/correlator/config.json`
- HTTP probes:
  - `/health`
  - `/ready`

Behavior:

1. Load the controller-generated config.
2. Validate the source and sink contract.
3. Poll the configured source file for appended JSON-line events.
4. Apply rule matching, time-window, and threshold logic.
5. Emit correlated events to the configured sink.
6. Report ready after the probe server is bound and config validation succeeds.

## Deployment Contract

The Deployment generated by the controller must mount:

- the config ConfigMap at `/etc/correlator`
- the source parent directory read-only
- the sink parent directory writable when `sink.type=file`

The current sink-only `EmptyDir` contract is replaced by path-aware mounts derived from the CR spec.

### Path Rules

To keep the contract bounded and repo-owned, the controller must reject paths outside the approved prefixes.

Allowed prefixes:

- source:
  - `/var/run/fos1/events/`
  - `/var/log/fos1/`
- sink:
  - `/var/log/fos1/`
  - `/var/log/correlator/`

If a path falls outside the allowed prefixes, reconciliation fails with an explicit validation error instead of creating a Deployment that looks valid but cannot be supported.

### Volume Strategy

- For file-based source and sink paths, mount the parent directories rather than the specific file path.
- Use `hostPath` volumes for the approved source and sink parent directories.
- Mount the source volume read-only.
- Mount the sink volume read-write.
- If source and sink share the same parent directory, reuse one volume rather than mounting duplicates.

This keeps the controller contract explicit and matches the repo-owned node-local file model.

## Build Contract

The repository must own a build path for `fos1/event-correlator:latest`.

Baseline requirement:

- add a repo-owned Docker build path for `cmd/event-correlator`
- ensure the resulting image contains `/usr/bin/event-correlator`

The controller continues to reference `fos1/event-correlator:latest`, but after this slice that reference is backed by a repo-owned artifact rather than an assumed external image.

## Testing And Verification

### Unit Tests

- `pkg/security/ids/correlation/runtime_test.go`
  - runtime reads file input and emits correlated output
  - invalid source/sink config fails clearly
- `pkg/security/ids/correlation/processor.go`
  - existing threshold/time-window tests remain the behavioral baseline

### Controller Tests

- `pkg/security/ids/correlation/controller_test.go`
  - Deployment contains repo-owned runtime command and config mount
  - Deployment mounts source parent directory read-only
  - Deployment mounts sink parent directory writable when `sink.type=file`
  - invalid source or sink path prefixes fail reconciliation
  - shared parent directory reuses a single volume when applicable

### Verification Commands

- `go test ./cmd/event-correlator ./pkg/security/ids/correlation/...`
- `go test ./pkg/security/ids/correlation/... ./pkg/apis/security/...`
- `make verify-mainline`

## Rollout Notes

This slice only puts the correlator on a real repo-owned runtime contract. It does not claim that live Suricata or Zeek production events are already flowing into that runtime. That broader proof remains a follow-on sprint after the controller and runtime contract are fully owned and testable.

## Acceptance Criteria

- The repository contains a buildable `event-correlator` command and owned image build path.
- The controller generates a Deployment that can read the configured source path and write the configured sink path under bounded prefixes.
- Invalid file paths fail clearly at reconciliation time.
- Runtime and controller tests cover the owned file-based contract.
- Docs do not over-claim live end-to-end event ingestion beyond the bounded runtime contract.
